## Executive Summary

This week marked a turning point in PlanExe's upstream evolution. The autonomously-running prompt_optimizer loop completed **10 merged PRs (#266-275)** focusing on schema consistency, prompt clarity, and lever deduplication. Simon ran multiple optimization iterations per day without manual intervention — the two-repo architecture (PlanExe + PlanExe-prompt-lab) is working as designed.

On the local model side, we ran GLM 4.7 Flash via LMStudio against two real farm planning scenarios (garden plan and chicken coop plan for Hampton, CT). Both runs cleared Phase 2 (PremiseAttackTask — previously a GLM failure point) but died at SelfAuditTask due to kernel panics under sustained GPU inference load. This is a hardware/driver issue, not a PlanExe bug — and it defines the boundary of what local models can currently handle on this hardware.

## PRs This Week

Six PRs merged to `PlanExeOrg/PlanExe` this week, all targeting `identify_potential_levers` via the prompt_optimizer loop:

**PR #268 — Remove doubled user prompt (B1)**
The user prompt was being injected twice into the continuation conversation, biasing the first call's output and contaminating the shared context. Fix reduced review format violations from 67 → 4.

**PR #270 — `message.content` vs `model_dump()` for assistant turn**
Root cause of the 33% unique lever name rate. `model_dump()` was producing Python `repr()` syntax instead of valid JSON in the continuation context — the LLM was receiving malformed prior output on calls 2 and 3. Fix: use `message.content` (verbatim string) with `model_dump_json()` fallback for function-calling models.

**PR #272 — Novelty-aware follow-up prompts**
Replaced bare `"more"` continuation prompts with explicit exclusion lists: `"Do NOT reuse any of these already-generated names: [...]"`. Also eliminated the `"25% faster scaling through..."` template leakage — gpt-5-nano was generating that phrase verbatim in 72/75 levers.

**PR #273 — Remove exemplar strings, make wrapper fields optional**
Removed concrete exemplar strings that anchored model output to specific patterns. Made `strategic_rationale` and `summary` optional in `DocumentDetails` to prevent hard validation failures when models skip those fields.

**PR #274 — Align Pydantic field descriptions with system prompt**
The `consequences` field description gave a plain-prose example while the system prompt required an I→S→S causal chain. Schema-sensitive models (gpt-4o-mini) were following the field description and ignoring the system prompt — causing 100% chain-format violations. Also fixed `options` field: "2-5 options" → "exactly 3".

**PR #275 — Fix consequences length and `review_lever` format**
Changed target from "150-300 words" to "3-5 sentences (~60-120 words)" — addresses haiku timeouts from oversized consequences. Added explicit trade-off requirement and prohibited review/critique contamination. Required dual-component format for `review_lever`.

**VoynichLabs/PlanExe2026 fork** synced to upstream `644fd59e` (all 27 commits including the above).

## Architecture

**Two-repo architecture validation**

The prompt_optimizer runs across two repos:
- `PlanExeOrg/PlanExe` — production code being optimized
- `PlanExeOrg/PlanExe-prompt-lab` — experiment history, analysis docs, baseline datasets

This separation is working as designed. Simon ran 6 optimization iterations this week without needing to touch the production repo structure — each iteration produced a numbered analysis directory with `insight_claude.md`, `insight_codex.md`, `code_claude.md`, `synthesis.md`, and `assessment.md`. The before/after comparisons are auditable.

**The autonomy question**

The loop is currently semi-autonomous: Simon reads the synthesis, decides which direction to implement, opens the PR, reviews the assessment verdict. The synthesis step (cross-agent reconciliation of Claude and Codex analyses) already produces a ranked list of directions. The implementation step still requires a human to select which direction to act on.

This week's data point: PR #276 (`fix/enforce-schema-contract`) was flagged as a regression before merge — the assessment caught it. The quality gate is working. Whether it's reliable enough to close the loop fully without a human checkpoint is still an open question.

**Local model boundary mapping**

GLM 4.7 Flash (via LMStudio) was tested this week against two farm planning scenarios. Key findings:
- `PremiseAttackTask` (Phase 2): passes with thinking disabled; fails with thinking enabled (schema echoing)
- `IdentifyDocumentsTask`: fails (JSON truncated at token limit)
- `SelfAuditTask` (20 items × ~300s): triggered GPU kernel panics under sustained inference load

This defines the current ceiling for local inference on this hardware. The SelfAuditTask failure is not a PlanExe bug — it's a macOS IOGPUFamily driver issue under prolonged M4 Pro GPU load.

### Local Model Validation — GLM 4.7 Flash

**Test Runs This Week:**
1. **Garden Plan** (Hampton, CT): Growing and selling produce — 146 output files
   - Model: GLM 4.7 Flash via LMStudio (local)
   - Result: Failed at SelfAuditTask (kernel panic in the planning pipeline)
   
2. **Chicken Coop Plan** (Hampton, CT): Egg production & management — 109 output files
   - Model: GLM 4.7 Flash via LMStudio (local)
   - Result: Failed at SelfAuditTask (same failure pattern)

**Findings:**
GLM 4.7 Flash handles the early phases well (ExecutiveSummary, DataCollection, Assumptions) but hits a wall at SelfAuditTask. The kernel panic suggests either:
- Memory pressure in the structured output phase
- Token limit collision within the audit prompting
- Schema validation failure on the output format

This validates why cloud-hosted model testing is important — we need models that can reliably complete the full pipeline, not just partial phases.

**What's Next:**
OpenRouter free-tier cascade testing against the same farm scenarios is a planned next step to determine whether cloud-hosted models avoid the SelfAuditTask failure. Results TBD.

### Model Selection Strategy

**Local Model Limitation:**
GLM 4.7 Flash's failure at SelfAuditTask is a hard blocker for end-to-end plan generation. This suggests that local models may need higher resource thresholds or different prompt structures to handle the full pipeline reliably.

**Next Validation:**
OpenRouter free-tier cloud models will be tested next against the same scenarios to see if they avoid this failure pattern. If they do, the cost-benefit case becomes clear:
- Local GLM 4.7 Flash: Free but incomplete (fails on full pipeline)
- OpenRouter free cascade: <$0.05 per plan but completes end-to-end
- Paid models (Claude/GPT): $0.50-2.00 per plan but guaranteed completion

**Ongoing Work:**
OpenRouter free-tier cascade testing is queued as the next validation phase once the kernel panic issue on local inference is understood.

**Failure mode classification:**
- `PremiseAttackTask` failure = model configuration issue (thinking=ON), not PlanExe bug
- `IdentifyDocumentsTask` failure = token/context limit, known local model ceiling
- `SelfAuditTask` failure = macOS GPU kernel panic (IOGPUFamily `completeMemory() prepare count underflow`), not PlanExe bug
